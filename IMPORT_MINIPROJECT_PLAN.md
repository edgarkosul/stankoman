# План реализации: Универсальный импорт товаров

## Цель
Собрать единый каркас импорта товарных данных с разделением ответственности:
- общий import core (оркестрация, логирование, сохранение, ошибки);
- парсеры источников/форматов (XML stream, HTML DOM и др.);
- адаптеры поставщиков (маппинг данных конкретного поставщика в внутренний формат).

## Границы минипроекта
- Входит: backend-архитектура импорта, перенос текущих парсеров в новый каркас, тесты, базовый операционный контроль.
- Не входит (первый этап): полный рефакторинг всех бизнес-правил каталога, сложная аналитика BI, массовая миграция старых данных.

## Архитектурные принципы
- Идемпотентность: повторный запуск не должен ломать данные.
- Потоковая обработка: без загрузки всего источника в память.
- Изоляция поставщика: вся специфика в adapter/profile слое.
- Расширяемость: добавление нового поставщика без изменений core.
- Наблюдаемость: каждый run прозрачно логируется и диагностируется.

## Этапы реализации

### Этап 0. Discovery и фиксация текущего состояния
1. Зафиксировать, как работают текущие 2 парсера: входы, выходы, ограничения.
2. Описать текущие точки входа импорта (CLI, UI, jobs).
3. Выписать обязательные поля внутренней модели товара.
4. Сформировать список рисков (кодировка, дубляжи, пропавшие товары, медиа).

**Результат:** документ с текущими контрактами и ограничениями.

### Этап 1. Контракты и DTO
1. Ввести единый внутренний `ProductPayload` (DTO/массив-форма). Минимум обязательных полей: `external_id`, `name`; остальные поля опциональны и валидируются режимом/опциями прогона.
2. Определить интерфейсы:
   - `SourceResolverInterface` (file/url/download/cache);
   - `RecordParserInterface` (yield сырых записей источника);
   - `SupplierAdapterInterface` (сырой record -> ProductPayload);
   - `ImportProcessorInterface` (валидировать/нормализовать/upsert).
3. Определить контракт ошибок (`ImportError`) и статусов run.

**Результат:** согласованные интерфейсы без привязки к конкретному поставщику.

### Этап 2. Import Run и логирование
1. Добавить сущность/таблицу `import_runs` (поставщик/`Source`, входной файл/URL, режим `partial_import/full_sync_authoritative`, флаги действий, статус, счетчики, время).
2. Добавить `import_run_errors` (уровень, код, сообщение, контекст record).
3. Реализовать сервис оркестрации run:
   - старт;
   - шаги пайплайна;
   - успешное завершение;
   - аварийное завершение.
4. Добавить порог отказа (например, остановка при X% критичных ошибок).

**Результат:** единый прозрачный жизненный цикл импорта.

### Этап 3. Источники и парсеры форматов
1. Реализовать загрузчик источников:
   - локальный файл;
   - URL с поддержкой timeout/retry;
   - опционально ETag/Last-Modified.
2. Реализовать `XmlStreamParser` на `XMLReader` с поддержкой:
   - выбора повторяющегося товарного узла;
   - детекта/конвертации кодировки в UTF-8.
3. Реализовать базовый `HtmlDomParser`:
   - обход карточек;
   - извлечение полей по selector/xpath-конфигу;
   - fallback-правила.
4. Реализовать `YmlStreamParser` (Yandex Market Language) на `XMLReader`:
   - чтение `<categories>` в карту `categoryId -> name`;
   - потоковое yield-извлечение `<offer>`.

**Результат:** общий слой парсинга форматов, независимый от поставщика.

### Этап 4. Нормализация и upsert
1. Реализовать нормализацию:
   - цена, валюта, остаток, bool/enum;
   - строковые чистки (trim, html entities, whitespace).
2. Определить правила идентификации товара:
   - `supplier + external_id` (или эквивалентный стабильный ключ).
3. Реализовать upsert-процесс батчами.
4. Определить политику “missing” (товары, исчезнувшие из источника): выполняется только в режиме `full_sync_authoritative` (finalize после прогона). В `partial_import` деактивация “missing” не выполняется.
5. Зафиксировать стратегию категорий: новые товары создаются в категории `Staged`, существующие обновляются без смены категорий (по умолчанию).

**Результат:** предсказуемое сохранение данных в БД.

### Этап 5. Медиа-пайплайн
1. Вынести загрузку изображений/документов в очередь.
2. Добавить дедупликацию по URL/хэшу.
3. Добавить ретраи и лимиты на размер/типы файлов.
4. Логировать ошибки медиа отдельно от основного импорта.

**Результат:** импорт товаров не блокируется медиа-операциями.

### Этап 6. Адаптеры поставщиков
1. Перенести текущий XML-парсер в новый `SupplierAdapter` + `XmlStreamParser`.
2. Перенести MetalMaster в `SupplierAdapter` + `HtmlDomParser` (или кастомный extractor при необходимости).
3. Для каждого адаптера оформить профиль:
   - маппинг полей;
   - правила категорий;
   - обработка характеристик;
   - fallback и дефолты.
4. Добавить `SupplierAdapter/Profile` `Yandex Market Feed`:
   - поддержка типа offer “упрощенный” (`<name>`) и `vendor.model` (склейка `<typePrefix>? + <vendor> + <model>`);
   - strict-валидация обязательных полей с `record-level` ошибками.

**Результат:** оба текущих поставщика работают через новый каркас.

### Этап 7. Точки запуска и эксплуатация
1. Команда запуска импорта (CLI) по поставщику/профилю с выбором режима (`partial_import`/`full_sync_authoritative`) и опций (create/update/check presence/media).
2. Интеграция с существующей админ-точкой запуска (Filament page): выбор `Source`, файла/URL, режима и опций прогона.
3. Базовое расписание (при необходимости) и защита от параллельных запусков.
4. Быстрый отчет по run: сколько обработано/обновлено/ошибок.

**Результат:** контролируемый запуск и эксплуатация.

### Этап 8. Тестирование и приемка
1. Набор feature-тестов на полный happy path для каждого адаптера.
2. Тесты на ошибки формата/кодировки/битые записи.
3. Тесты идемпотентности повторного запуска.
4. Тесты на политику отсутствующих товаров и медиа-ретраи.

**Результат:** подтвержденная стабильность ключевых сценариев.

## Очередность внедрения
1. Этапы 0-2.
2. Этапы 3-4.
3. Этап 6 (перенос 2 текущих парсеров).
4. Этапы 5 и 7.
5. Этап 8 (финальная приемка).

## Definition of Done (минипроекта)
- Оба текущих поставщика работают через единый import core.
- Добавление нового поставщика сводится к новому adapter/profile без правок core.
- Импорт прозрачно логируется на уровне run и ошибок.
- Повторный импорт идемпотентен.
- Есть автоматические тесты на критичные сценарии.
